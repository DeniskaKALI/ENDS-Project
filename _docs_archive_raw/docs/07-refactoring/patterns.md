# Паттерны персистентности: Identity Map

## Что это такое

**Identity Map** (карта идентичностей) — паттерн из *Patterns of Enterprise Application Architecture* (Martin Fowler). Он обеспечивает правило: **в рамках одного контекста персистентности для каждого первичного ключа существует не более одного экземпляра сущности в памяти**.

Повторный запрос «дай сущность с `id = X`» не создаёт второй Java-объект — возвращается тот же managed-экземпляр, что уже есть в карте.

## Как Hibernate реализует Identity Map

В JPA/Hibernate роль Identity Map выполняет **Persistence Context** (контекст персистентности) — область памяти, которой управляет `EntityManager` (в Spring — обычно один контекст на транзакцию).

Внутри контекста Hibernate хранит структуру вида:

```text
Map<Идентификатор, Managed Entity>
```

При вызове `entityManager.find(Item.class, id)` или `repository.findById(id)`:

1. Hibernate ищет сущность в Persistence Context по первичному ключу.
2. Если объект уже есть — возвращает **существующую ссылку** (без повторного SELECT или с минимальной работой).
3. Если объекта нет — загружает строку из БД, помещает в контекст и регистрирует в карте.

Проверка в коде: `first == second` для двух загрузок в **одной** транзакции даёт `true`.

Демонстрация в проекте:

| Артефакт | Назначение |
|----------|------------|
| `ItemService#getItemWithIdentityCheck(UUID)` | Два `findById` в одном `@Transactional` методе + лог `==` |
| `IdentityMapTest` | Интеграционный тест с `EntityManager` и `assertSame()` |

## Почему это важно для консистентности данных

1. **Единое состояние в памяти** — если изменить поле у `first`, то `second` видит то же значение, потому что это один объект. Нет расхождения «две копии одной строки БД».
2. **Один flush на сущность** — Hibernate отслеживает изменения dirty-checking на единственном экземпляре; не возникает конфликта «какая копия правильная».
3. **Предсказуемое поведение коллекций и связей** — навигация по `@ManyToOne` / `@OneToMany` в рамках сессии согласована с тем же контекстом.

Без Identity Map повторные загрузки могли бы порождать несколько detached-копий с одним `id`, что усложняет обновления и повышает риск перезаписи данных.

## Когда Identity Map сбрасывается

Карта идентичностей живёт **только пока активен Persistence Context**. Контекст заканчивается, когда:

| Событие | Эффект |
|---------|--------|
| **Завершение транзакции** (`commit` / `rollback`) | Контекст закрывается, managed-объекты становятся detached (или отбрасываются) |
| **`entityManager.clear()`** | Все сущности отсоединяются из контекста; карта очищается |
| **`entityManager.detach(entity)`** | Конкретная сущность удаляется из карты |
| **Закрытие `EntityManager`** (конец request/session в некоторых конфигурациях) | Аналогично завершению жизненного цикла контекста |

После этого следующий `findById(id)` в **новой** транзакции создаст **новый** managed-экземпляр (другая ссылка в JVM), хотя данные в БД те же. Оператор `==` между объектами из разных транзакций, как правило, даст `false`.

В Spring типичная модель:

```text
@Transactional метод  →  открытие транзакции  →  Persistence Context + Identity Map
commit/rollback       →  закрытие транзакции  →  контекст уничтожен
```

Поэтому демонстрацию Identity Map нужно выполнять **внутри одного** `@Transactional` границы (как в `getItemWithIdentityCheck` и в `IdentityMapTest`, где `@DataJpaTest` оборачивает каждый тест в транзакцию).

# Паттерны персистентности: Lazy Load

## Что это такое

**Lazy Load** (ленивая загрузка) — связанные сущности и коллекции не подгружаются из БД до первого обращения к полю в коде. В JPA это задаётся `fetch = FetchType.LAZY` (для `@OneToMany` и `@ManyToMany` — значение по умолчанию).

Прокси или «незагруженная» коллекция существует в памяти, но SQL выполняется только при вызове геттера, и только если открыт Persistence Context (активна транзакция).

## Настройка FetchType в myShelf

| Связь | FetchType | Обоснование |
|-------|-----------|-------------|
| `User.items` | `LAZY` | У пользователя может быть сотни вещей; не грузить всё при каждом `findById(User)` |
| `User.outfits` | `LAZY` | То же для списка образов |
| `Outfit.slots` | `EAGER` | Образ без слотов бесполезен; слоты — неотъемлемая часть агрегата при отображении |
| `OutfitSlot.item` | `LAZY` | Вещь в слоте опциональна; `itemId` для DTO часто достаточно без полной загрузки `Item` |
| `Item.user`, `Outfit.user` | `LAZY` | Владелец редко нужен в каждом запросе списка вещей/образов |

## Почему для `@OneToMany` используется LAZY

1. **Объём данных** — коллекция может быть большой; жадная загрузка всех дочерних строк при каждом чтении родителя даёт лишний трафик и память.
2. **Риск Cartesian Product** — `JOIN` родителя с большой коллекцией раздувает результат SQL.
3. **Принцип отложенной работы** — загружаем только то, что реально используем в сценарии.

Для `User.items` и `User.outfits` LAZY означает: при `userRepository.findById(id)` Hibernate **не** выполняет `SELECT` по таблицам `items` / `outfits`, пока код не вызовет `user.getItems()` или `user.getOutfits()`.

## Когда нужен EAGER

`FetchType.EAGER` уместен, когда связь **всегда** нужна вместе с родителем и её отсутствие ломает бизнес-смысл:

- `Outfit.slots` — образ без слотов не имеет смысла в UI и API; слоты загружаются вместе с образом.

EAGER не отменяет необходимость `@Transactional` при доступе к другим LAZY-полям (например, `OutfitSlot.item`). Для точечной загрузки графа связей предпочтительны `@EntityGraph` и `JOIN FETCH`, а не глобальный EAGER на всех связях.

## LazyInitializationException

Ошибка возникает, если обратиться к LAZY-полю **после** закрытия Persistence Context (конец транзакции, выход из `@Transactional` метода, сериализация entity в JSON без DTO).

Решения в проекте:

- Держать чтение связей внутри `@Transactional(readOnly = true)` сервисного метода.
- Использовать DTO-мапперы, а не отдавать entity наружу.
- Явно подгружать нужный граф до маппинга (см. ниже).

## Как `@EntityGraph` решает проблему N+1

**N+1** — один запрос за список родителей и по одному запросу на каждую связь при обходе коллекции в цикле.

Пример: 50 вещей пользователя, у каждой в коде вызывается `item.getUser().getEmail()` → 1 + 50 запросов.

`@EntityGraph(attributePaths = {"user"})` на методе репозитория указывает Hibernate сформировать запрос с **join fetch** (или отдельный batched fetch) для поля `user`:

```java
@EntityGraph(attributePaths = {"user"})
List<Item> findByUser_Id(UUID userId);
```

В `ItemService#getItemsByUserId` используется этот метод: владелец подгружается **одним** запросом вместе со списком вещей.

Аналог для образа — JPQL с `JOIN FETCH`:

```java
@Query("SELECT o FROM Outfit o LEFT JOIN FETCH o.slots WHERE o.id = :id")
Optional<Outfit> findWithSlots(@Param("id") UUID id);
```

`OutfitService#getOutfitDetails` вызывает `findWithSlots`, поэтому `outfit.getSlots()` и маппинг в `OutfitDTO` не вызывают `LazyInitializationException` и не порождают отдельный SELECT на каждый слот при первичной загрузке образа.

| Подход | Когда использовать |
|--------|-------------------|
| `LAZY` по умолчанию | Большинство `@OneToMany` / `@ManyToOne` |
| `EAGER` на агрегате | Связь обязательна всегда (`Outfit.slots`) |
| `@EntityGraph` | Нужна LAZY-модель, но в конкретном use-case — join к `user` |
| `JOIN FETCH` в `@Query` | Явный контроль графа (`findWithSlots`) |

## Артефакты в коде

| Артефакт | Назначение |
|----------|------------|
| `ItemRepository#findByUser_Id` | Entity Graph для владельца вещи |
| `OutfitRepository#findWithSlots` | JOIN FETCH слотов образа |
| `OutfitService#getOutfitDetails` | Детали образа без LazyInitializationException на `slots` |
