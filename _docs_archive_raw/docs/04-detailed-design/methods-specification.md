# СПЕЦИФИКАЦИЯ МЕТОДОВ КЛЮЧЕВЫХ КОМПОНЕНТОВ СИСТЕМЫ

Документ описывает сигнатуры и поведение основных методов серверной и клиентской частей приложения «Моя полка». Спецификация сгруппирована по архитектурным слоям в соответствии с паттерном PCMEF и отражает бизнес-логику прецедентов UC-005, UC-013, UC-020 и UC-021.

## 1. Серверный уровень (Java / Spring Boot)

### 1.1. Модуль аутентификации и управления аккаунтом

**1.1.1. `registerUser`**
```java
public ResponseEntity<AuthResponse> registerUser(@Valid @RequestBody UserRegistrationDTO request)
```
*Описание:* Метод принимает данные для создания новой учётной записи, выполняет проверку уникальности адреса электронной почты и применяет криптографическое хеширование пароля перед сохранением сущности в базу данных. При успешной регистрации система автоматически генерирует JWT-токен доступа, инициализирует запись пользовательских настроек со значениями по умолчанию и возвращает клиенту ответ, содержащий токен и базовую информацию о профиле. В случае нарушения ограничений валидации или попытки регистрации на уже занятый email метод прерывает выполнение и возвращает соответствующий HTTP-статус ошибки.

**1.1.2. `changePassword`**
```java
public ResponseEntity<Void> changePassword(@Valid @RequestBody PasswordChangeDTO dto, Authentication authentication)
```
*Описание:* Функция выполняет безопасную смену пароля авторизованного пользователя. Перед применением новых учётных данных метод извлекает текущий хеш пароля из базы и сравнивает его с предоставленным значением, предотвращая несанкционированное изменение. После подтверждения подлинности новый пароль проходит валидацию на соответствие политикам сложности, хешируется алгоритмом BCrypt и сохраняется в поле пользователя. Операция выполняется в рамках транзакции, гарантируя атомарность изменения, а клиент получает уведомление об успешном обновлении учётных данных.

**1.1.3. `updateProfile`**
```java
public ResponseEntity<UserProfileDTO> updateProfile(@Valid @RequestBody UpdateProfileDTO dto, Authentication authentication)
```
*Описание:* Метод отвечает за обновление публичных данных профиля, включая отображаемое имя и ссылку на аватар. Сервисный слой загружает текущую сущность пользователя, применяет изменения только к разрешённым полям и сохраняет результат. При обновлении аватара система дополнительно проверяет формат и доступность внешнего ресурса изображения. Возвращаемый объект содержит актуальные данные профиля, что позволяет клиентскому интерфейсу немедленно отобразить изменения без дополнительного запроса.

### 1.2. Модуль управления гардеробом (Вещи и Образы)

**1.2.1. `getUserItems`**
```java
public ResponseEntity<List<ItemDTO>> getUserItems(Authentication authentication)
```
*Описание:* Функция формирует и возвращает полный каталог вещей, принадлежащих текущему авторизованному пользователю. Метод извлекает идентификатор пользователя из контекста безопасности Spring Security, передаёт его в репозиторий для выборки записей и преобразует сущности базы данных в DTO-объекты, исключая внутренние технические поля. Если у пользователя отсутствуют добавленные вещи, метод возвращает пустую коллекцию вместо ошибки, обеспечивая корректную работу интерфейса при первом запуске.

**1.2.2. `createItem`**
```java
public ResponseEntity<Void> createItem(@Valid @RequestBody ItemDTO dto, Authentication authentication)
```
*Описание:* Метод создаёт новую карточку предмета гардероба и привязывает её к владельцу аккаунта. Перед сохранением выполняется валидация обязательных полей, включая наличие названия и категории, а также проверка корректности перечисления сезонности. Система автоматически заполняет метаданные создания и обновления, генерирует UUID идентификатор и сохраняет запись в таблицу `items`. Успешное выполнение возвращает статус `201 Created`, сигнализируя клиенту о необходимости обновления локального списка.

**1.2.3. `createOutfit`**
```java
public ResponseEntity<OutfitDTO> createOutfit(@Valid @RequestBody OutfitDTO dto, Authentication authentication)
```
*Описание:* Функция реализует создание структурированного образа с поддержкой слотовой архитектуры. Сервисный слой принимает коллекцию слотов, каждый из которых содержит тип категории и ссылку на вещь, и выполняет кросс-валидацию: система убеждается, что категория выбранной вещи строго соответствует типу слота, предотвращая логические ошибки компоновки. После проверки консистентности метод сохраняет сущность образа и связанные записи слотов в единой транзакции, возвращая клиенту полную модель с присвоенными идентификаторами.

**1.2.4. `deleteItem`**
```java
public ResponseEntity<Void> deleteItem(@PathVariable UUID id, Authentication authentication)
```
*Описание:* Метод выполняет удаление вещи из гардероба пользователя с учётом зависимостей в образах. Перед физическим удалением записи система проверяет право собственности, чтобы исключить доступ к чужим данным. При наличии образа, использующего удаляемую вещь, система не прерывает операцию, а обновляет соответствующие слоты, устанавливая значение ссылки на вещь в `NULL`, что сохраняет структуру образа в виде шаблона. Транзакционный подход гарантирует отсутствие осиротевших записей в таблице слотов.

## 2. Клиентский уровень (Kotlin / Android)

### 2.1. Сетевой слой (Retrofit API)

**2.1.1. `getItems`**
```kotlin
@GET("/api/items") 
suspend fun getItems(@Header("Authorization") token: String): List<ItemDTO>
```
*Описание:* Функция отправляет асинхронный GET-запрос к серверному эндпоинту для получения актуального списка вещей. Метод требует передачи JWT-токена в заголовке авторизации, что обеспечивает защиту данных на уровне транспорта. При успешном ответе сервера Retrofit автоматически десериализует JSON-массив в коллекцию DTO-объектов, которые затем передаются в слой репозиториев для обработки и сохранения в локальный кэш. Обработка сетевых исключений вынесена в перехватчики, стандартизирующие коды ошибок.

**2.1.2. `createItem`**
```kotlin
@POST("/api/items") 
suspend fun createItem(@Header("Authorization") token: String, @Body item: ItemDTO): Response<Void>
```
*Описание:* Метод передаёт данные новой вещи на сервер для персистентного сохранения. Функция оборачивает вызов в объект `Response`, позволяя клиентскому коду анализировать HTTP-статусы без выбрасывания исключений при ошибках валидации. При получении статуса 201 репозиторий помечает локальную запись как синхронизированную, а при статусах 4xx запускает процедуру обработки ошибок валидации, возвращая пользователю конкретные причины отказа в сохранении.

### 2.2. Слой локального хранения (Room DAO)

**2.2.1. `getItemsByUser`**
```kotlin
@Query("SELECT * FROM items WHERE userId = :userId ORDER BY createdAt DESC") 
suspend fun getItemsByUser(userId: String): List<ItemLocal>
```
*Описание:* Функция выполняет асинхронную выборку всех предметов гардероба конкретного пользователя из локальной базы данных SQLite. Запрос включает сортировку по дате создания в обратном порядке, что обеспечивает отображение недавно добавленных вещей в начале списка интерфейса. Метод работает исключительно с локальным хранилищем, гарантируя мгновенный отклик даже при отсутствии сетевого соединения, и возвращает список сущностей Room, готовых к прямому использованию в адаптерах или потоках данных.

**2.2.2. `upsertItem`**
```kotlin
@Insert(onConflict = OnConflictStrategy.REPLACE) 
suspend fun upsertItem(item: ItemLocal)
```
*Описание:* Метод реализует логику вставки или обновления записи вещи в зависимости от наличия совпадающего первичного ключа. Стратегия `REPLACE` автоматически перезаписывает существующую строку при конфликте идентификаторов, что критически важно для процесса синхронизации, когда серверные данные должны замещать устаревшие локальные версии. Функция выполняется в корутине, не блокируя главный поток, и возвращает управление сразу после фиксации транзакции в базе данных.

### 2.3. Репозитории и синхронизация

**2.3.1. `getItemsFlow`**
```kotlin
fun getItemsFlow(userId: String): Flow<List<ItemLocal>>
```
*Описание:* Функция возвращает реактивный поток данных, который автоматически эмитирует обновлённый список вещей при любом изменении в соответствующей таблице Room. Метод выступает единой точкой доступа для ViewModel, полностью абстрагируя источник данных и позволяя интерфейсу подписываться на изменения без ручного опроса. Поток конфигурируется с операторами `buffer` и `conflated`, чтобы гарантировать доставку только актуального состояния списка и избегать накопления устаревших событий при быстрой прокрутке или частых обновлениях.

**2.3.2. `syncItemsWithServer`**
```kotlin
suspend fun syncItemsWithServer(userId: String, token: String)
```
*Описание:* Метод оркестрирует процесс двусторонней синхронизации данных между устройством и облачным сервером. Сначала функция собирает все локальные записи с флагом `isDirty`, отправляет их на сервер через Retrofit и обрабатывает ответы, помечая успешные операции как синхронизированные. Затем выполняется загрузка полного актуального списка с сервера, который сравнивается с локальным кэшем по временным меткам `updated_at`, после чего применяются недостающие или изменённые записи. Ошибки сети перехватываются и логируются, а процесс ставится в очередь WorkManager для повторного запуска при восстановлении соединения.

### 2.4. Управление состоянием (ViewModel)

**2.4.1. `loadItems`**
```kotlin
fun loadItems()
```
*Описание:* Функция инициирует загрузку каталога вещей, запуская сбор данных из репозитория и преобразуя результат в состояния UI. Метод обрабатывает три сценария: отображение индикатора загрузки во время запроса, рендеринг списка при успехе и показ заглушки с сообщением об ошибке при сбое. Все состояния публикуются в `StateFlow`, который наблюдается экраном композиции, обеспечивая реактивное обновление интерфейса без прямых обращений к Android-компонентам из слоя бизнес-логики.

**2.4.2. `toggleAppTheme`**
```kotlin
fun toggleAppTheme()
```
*Описание:* Метод переключает визуальное оформление приложения между светлой и тёмной палитрами. Функция обновляет значение в локальном хранилище настроек, уведомляет диспетчер тем Android о необходимости применения нового ресурса и отправляет асинхронный запрос на сервер для сохранения предпочтения в профиле пользователя. Переключение выполняется мгновенно на клиенте, а синхронизация с облаком происходит в фоновом режиме, не блокируя взаимодействие с интерфейсом.

## 3. Заключение
Представленные методы покрывают полный цикл взаимодействия клиента и сервера, от аутентификации до управления сложными структурами образов. Серверная часть акцентирует внимание на валидации, транзакционной целостности и безопасности через Spring Security, тогда как клиентская часть оптимизирована под реактивное программирование, офлайн-доступ и плавную синхронизацию. Сигнатуры отражают современные практики Kotlin Coroutines, Flow и Spring Data, обеспечивая модульность и тестируемость каждого компонента в соответствии с требованиями проекта.
