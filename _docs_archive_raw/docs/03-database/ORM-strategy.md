# СТРАТЕГИЯ ОБЪЕКТНО-РЕЛЯЦИОННОГО ОТОБРАЖЕНИЯ (ORM)

## 1. Введение и технологический стек
Стратегия маппинга объектов предметной области на реляционную модель базы данных базируется на использовании фреймворка **Spring Data JPA** (реализация **Hibernate**) на стороне сервера и **Room Persistence Library** на стороне мобильного клиента. Основная цель данной стратегии — обеспечить прозрачную работу с данными, сохраняя при этом четкое разделение ответственности между слоями архитектуры PCMEF.

На серверной части Java-сущности (`Entity`) напрямую отображаются на таблицы **PostgreSQL**. Для управления транзакциями и жизненным циклом объектов используется декларативный подход через аннотации JPA. На клиентской части Android-модели данных синхронизируются с локальной **SQLite** базой через Data Access Objects (**DAO**), обеспечивая работу в офлайн-режиме.

## 2. Общие принципы маппинга
- **Именование полей:** В Java-классах следует стандарт `camelCase`, в то время как в базе данных используется `snake_case`. Для автоматического преобразования имен колонок применяется стратегия `PhysicalNamingStrategy` (или явное указание имени в аннотации `@Column`), что позволяет избежать ошибок при чтении кода.
- **Идентификаторы:** Каждая сущность наследуется от базового класса или интерфейса, содержащего идентификатор типа `UUID`. Использование UUID вместо автоинкремента (`Long`) выбрано для упрощения генерации идентификаторов на клиенте перед отправкой на сервер и повышения безопасности (предотвращение перебора ID сущностей).
- **Разделение моделей:** Важным принципом является разделение **Entity-моделей** (используемых внутри сервера и БД) и **DTO-моделей** (Data Transfer Objects), которые передаются через REST API. Сущности никогда не возвращаются клиенту напрямую, чтобы избежать утечки внутренней структуры базы данных и циклических ссылок при сериализации JSON.

## 3. Описание маппинга ключевых сущностей

### Пользователь и Настройки (`User` & `UserSettings`)
1. Сущность `User` является корневым агрегатом. Она отображается на таблицу `users`. Поле `passwordHash` маппится с атрибутом `nullable = false` и исключается из сериализации JSON с помощью аннотации `@JsonIgnore`, так как пароль никогда не передается клиенту.
2. Связь с `UserSettings` реализована как `One-to-One`. В базе данных это обеспечено тем, что первичный ключ таблицы `user_settings` является одновременно внешним ключом на `users`. В Java это отражается аннотацией `@OneToOne` с каскадным удалением (`CascadeType.ALL`), гарантируя, что настройки не останутся "сиротами" при удалении пользователя.

### Вещи гардероба (`Item`)
1. Сущность `Item` отображается на таблицу `items`. Связь с пользователем является `Many-to-One` (много вещей у одного пользователя). Аннотация `@ManyToOne` на поле `owner` (`User`) позволяет быстро загружать владельца вещи.
2. Для оптимизации выборки используется стратегия загрузки `FetchType.LAZY`. Это означает, что при запросе списка вещей данные о пользователе не подгружаются автоматически, что экономит ресурсы памяти. Однако при удалении пользователя (`CascadeType.REMOVE`) все связанные вещи удаляются автоматически силами Hibernate.

### Образы и Слоты (`Outfit` & `OutfitSlot`)
Это наиболее сложная часть модели. В базе данных связь между образом и вещью реализована через промежуточную таблицу `outfit_slots`, так как нам нужно хранить дополнительную информацию о роли вещи в образе (тип слота).
- Следовательно, в Java мы не используем стандартную аннотацию `@ManyToMany`. Вместо этого мы создаем полноценную сущность `OutfitSlot`.
- Сущность `Outfit` имеет коллекцию слотов (`List<OutfitSlot>`) с аннотацией `@OneToMany(mappedBy = "outfit", cascade = CascadeType.ALL, orphanRemoval = true)`. Параметр `orphanRemoval = true` критически важен: если слот удаляется из коллекции в Java, он автоматически удаляется из базы данных.
- Сущность `OutfitSlot` содержит ссылку на `Item` (`@ManyToOne`). Здесь настроено поведение `optional = true`, что соответствует допустимости `NULL`-значения в базе данных (слот может быть пустым).

## 4. Работа с перечислениями (Enums)
Для полей `category` и `season` в сущностях `Item` и `Outfit`, а также `slot_type` в `OutfitSlot`, используется аннотация `@Enumerated(EnumType.STRING)`.

Это решение предпочтительнее использования `EnumType.ORDINAL`, так как в базе данных хранятся читаемые строковые значения (`'TOP'`, `'WINTER'` и т.д.). Это делает данные в БД понятными для человека и защищает приложение от поломок при изменении порядка элементов в Java-перечислении в будущем.

## 5. Стратегия синхронизации (Client-Server)
Поскольку приложение поддерживает офлайн-режим, стратегия ORM на клиенте (**Room**) должна зеркально отражать серверную модель, но с учетом локальной специфики.
- Клиентские сущности (`Room Entities`) имеют те же поля, что и серверные, но дополнительно содержат поле `isDirty` (или `syncStatus`). Когда пользователь создает вещь локально, она сохраняется в Room с флагом "требует синхронизации". Фоновый процесс (`WorkManager`) считывает такие записи, отправляет их на сервер через `Retrofit`, и в случае успеха обновляет статус записи.
- При получении данных с сервера (например, списка образов) клиентская ORM-стратегия предполагает полную замену локальных данных серверными (стратегия `"Remote Wins"` для конфликтующих полей), чтобы гарантировать целостность данных, если пользователь заходил с другого устройства.

## 6. Валидация и ограничения
На уровне ORM используются аннотации валидации **Bean Validation** (`javax.validation` или `jakarta.validation`).
- `@NotNull` и `@Size` применяются к полям сущностей для обеспечения целостности данных до попадания в БД.
- Уникальность `email` пользователя гарантируется не только аннотацией `@Column(unique = true)`, но и дублируется ограничением на уровне **PostgreSQL** (как указано в DDL), что защищает от гонок состояний (`race conditions`).

## 7. Identity Map и Persistence Context

Hibernate реализует паттерн **Identity Map** через **Persistence Context**: в рамках одной транзакции повторная загрузка сущности по тому же первичному ключу возвращает тот же managed-экземпляр в памяти (`first == second`). Это обеспечивает консистентность изменений в JVM до `flush`/`commit`.

Подробное описание, демонстрация в коде (`ItemService#getItemWithIdentityCheck`) и интеграционный тест `IdentityMapTest` — в документе [Паттерны персистентности](../07-refactoring/patterns.md).

## 8. Lazy Load и контролируемая загрузка связей

Связи `@OneToMany` у `User` (`items`, `outfits`) объявлены с `FetchType.LAZY`, чтобы не подтягивать весь гардероб при каждой загрузке пользователя. У `Outfit.slots` задан `EAGER`, так как образ без слотов не используется в предметной области. Для `OutfitSlot.item` оставлен `LAZY`.

Точечная подгрузка без N+1:

- `ItemRepository.findByUser_Id` — `@EntityGraph(attributePaths = {"user"})`;
- `OutfitRepository.findWithSlots` — `LEFT JOIN FETCH o.slots`;
- `OutfitService.getOutfitDetails` — использует `findWithSlots` перед маппингом в DTO.

Подробнее: [Lazy Load и EntityGraph](../07-refactoring/patterns.md#паттерны-персистентности-lazy-load).
