# АРХИТЕКТУРНЫЕ РЕШЕНИЯ (ADR)

## ADR-001: Выбор клиент-серверной архитектуры
**Статус:** Принято

**Контекст:**
Приложение должно обеспечивать доступ к данным гардероба с мобильных устройств. Возникает необходимость в сохранении данных при переустановке приложения и потенциальном расширении на другие платформы (например, веб-версию).

**Решение:** Использовать клиент-серверную архитектуру.
1. **Клиент:** Мобильное приложение (Android Native / Kotlin), отвечающее за UI и локальное кэширование.
2. **Сервер:** Удаленное приложение (Java / Spring Boot), выступающее единственным источником истины (Source of Truth) для постоянных данных и централизованной аутентификации.

**Последствия:**
1. **Плюсы:** Данные защищены и доступны с разных устройств (в перспективе), сервер берет на себя тяжелые вычисления и валидацию.
2. **Минусы:** Приложение требует наличия сети для первичной синхронизации. Требуется реализация механизма разрешения конфликтов при синхронизации.

## ADR-002: Применение паттерна PCMEF
**Статус:** Принято

**Контекст:**
Курсовой проект требует демонстрации навыков структурирования кода, обеспечения тестируемости и разделения ответственности. Прямое обращение UI к базе данных или сети затрудняет поддержку кода.

**Решение:** Внедрить адаптированную версию паттерна PCMEF:
1. **Presentation:** Отвечает только за отображение (UI / UX).
2. **Control / State:** Управляет состоянием экрана (ViewModel / BLoC).
3. **Mediator (Client) / Mediator (Server):** Содержит бизнес-логику и правила (Use Cases / Services).
4. **Entity:** Доменные модели данных.
5. **Foundation:** Инфраструктурный слой (Room, Retrofit, JPA Repositories).

**Последствия:**
1. **Плюсы:** Строгое разделение слоев, высокая тестируемость (можно тестировать Mediator без UI и БД), код становится модульным.
2. **Минусы:** Увеличение количества файлов и классов (boilerplate), необходимость создания интерфейсов для взаимодействия между слоями.

## ADR-003: Стратегия работы с данными (Offline-First)
**Статус:** Принято

**Контекст:**
Пользователи могут использовать приложение в местах с нестабильным интернет-соединением (например, в магазине одежды или в гардеробной). Приложение не должно зависать или падать при отсутствии сети.

**Решение:** Реализовать стратегию Offline-First.
1. Локальная база данных (Room на клиенте) является главным источником данных для UI.
2. При выполнении действий (добавление вещи) изменения сначала сохраняются локально.
3. Фоновый сервис (WorkManager) синхронизирует изменения с сервером при появлении подключения.

**Последствия:**
1. **Плюсы:** Мгновенный отклик интерфейса, работа в офлайн-режиме.
2. **Минусы:** Усложнение логики синхронизации. Необходимо обрабатывать ситуации, когда данные на сервере изменились во время отсутствия связи у клиента.

## ADR-004: Структура данных образа (Концепция Слотов)
**Статус:** Принято

**Контекст:**
При создании образа пользователи могут хаотично добавлять вещи. Чтобы образ был структурированным и визуально понятным, необходимо разделить предметы по категориям (голова, тело, ноги, обувь).

**Решение:** Реализовать модель `Outfit`, состоящую из коллекции `OutfitSlot`.
1. Каждый `OutfitSlot` имеет жестко заданный тип (Category: `HEADWEAR`, `TOP`, `BOTTOM` и т.д.).
2. В слот можно добавить только `Item`, чья категория совпадает с типом слота.

**Последствия:**
1. **Плюсы:** Гарантия целостности данных (нельзя добавить обувь в слот для шляпы), упрощенный UI конструктора (пользователь видит пустые "ячейки" для одежды).
2. **Минусы:** Более сложная модель базы данных (связь One-to-Many с дополнительной сущностью `Slot`), необходимость валидации на сервере при сохранении образа.

## ADR-005: Выбор протокола взаимодействия API
**Статус:** Принято

**Контекст:**
Необходимо обеспечить обмен данными между клиентом (Kotlin) и сервером (Java). Требуется стандартизация и документирование интерфейса.

**Решение:** Использовать REST API с форматом данных JSON.
1. Аутентификация через заголовок `Authorization: Bearer <token>`.
2. Документация генерируется автоматически через OpenAPI (Swagger).

**Последствия:**
1. **Плюсы:** Стандарт де-факто для веба, простота отладки, легкая генерация клиентских SDK (Retrofit).
2. **Минусы:** Over-fetching (получение лишних данных) в некоторых сценариях по сравнению с GraphQL, однако для MVP это не критично.

## ADR-006: Безопасность и Аутентификация
**Статус:** Принято

**Контекст:**
Гардероб содержит личные фотографии и данные пользователя. Доступ к аккаунту должен быть защищен, а сервер не должен хранить состояние сессий в памяти (stateless).

**Решение:** Использовать JWT (JSON Web Tokens).
1. При логине сервер выдает токен.
2. Клиент хранит токен в защищенном хранилище.
3. Каждый запрос к API сопровождается токеном.
4. Серверная часть защищена Spring Security.

**Последствия:**
1. **Плюсы:** Масштабируемость (серверу не нужно хранить сессии в БД), безопасность передачи данных.
2. **Минусы:** Нельзя мгновенно отозвать доступ пользователю без реализации "черных списков" токенов (для курсового проекта допустимо упрощение: доступ отзывается только после истечения срока действия токена).

## ADR-007: Выбор технологий хранения данных
**Статус:** Принято

**Контекст:**
Требуется надежное хранение структурных данных и их связей (пользователь → вещи → образы).

**Решение:**
1. **Сервер:** PostgreSQL + Spring Data JPA. Реляционная модель идеально подходит для строгих связей между сущностями.
2. **Клиент:** Room Database (обертка над SQLite).

**Последствия:**
1. **Плюсы:** ACID-транзакции, целостность данных (Foreign Keys), мощные средства запросов.
2. **Минусы:** Фиксированная схема данных (Schema), изменения структуры требуют миграций (Room / DB Migrations).
